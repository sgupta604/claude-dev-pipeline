## 1. What This Is and What Changed

`boreas-web` is the planning brain of the Boreas platform — a web app where users draw farm field polygons, see real wind data, compute spray drift, generate wind-compensated flight paths, and export waypoints to their drones.

**v2 resolves the following contradictions from v1:**

| Decision | v1 said | Research said | **v2 locks in** |
|---|---|---|---|
| Weather pipeline (Phase 1) | Herbie as primary | Open-Meteo for MVP | **Shyft/Folkweather OGC EDR (reuse mobile knowledge)** |
| Path planning | Fields2Cover | Pure Python/Shapely | **Pure Python/Shapely for MVP (~300–500 lines)** |
| Phase 1 drift model | "Tier 1 lookup table in TypeScript" | Lookup table pre-computed by Rust engine | **Simple ballistic model in TypeScript for Phase 1; Rust lookup table in Phase 3** |
| 3D Dome timing | Phase 4 | "Primary visualization surface" | **Phase 4 — validate planning tools on 2D map first** |
| Fields2Cover deployment | Referenced without qualification | 2–3 GB Docker images, 15–45 min builds | **Not used. Pure Python/Shapely instead. Rust migration later.** |

### Key principles

1. **Server does I/O and shared computation. Client does rendering.** Nothing that needs 60fps touches the server.

4. **Ship fast, validate with users, then optimize.** TypeScript ballistic drift in Phase 1 → Python Lagrangian in Phase 2 → Rust/PyO3 + WASM in Phase 3.

## 3. Wind Data Pipeline

### Phase 1: Shyft/Folkweather OGC EDR (reuse mobile knowledge)

Same providers, same quirks, same format as the Flutter mobile app. The FastAPI server proxies and caches.

**Shyft EDR (primary — GFS, 25km global)**
- Base: `https://ogc.shyftwx.com/ogc/edr/collections`
- Auth: `apikey` query parameter
- Collections: `GFS_isobaric` (pressure levels), `GFS_height-above-ground_10` (surface 10m wind)
- Params: `u-component-of-wind`, `v-component-of-wind`
- ⚠️ Area queries REQUIRE `f=CoverageJSON_MultiPointSeries` (plain `f=CoverageJSON` → HTTP 400)
- ⚠️ Uses a non-obvious `composite` axis format in MultiPointSeries responses
- ⚠️ 1000 hPa returns Internal Server Error — use surface collection instead

**Folkweather EDR (fallback — HRRR 3km US-only)**
- Base: `https://folkweather.com/edr/collections`
- Auth: none
- Collections: `hrrr-isobaric`, `gfs-isobaric-latest`, `hrrr-height-agl` (surface)
- Params: `UGRD`, `VGRD`
- ⚠️ Uses 0–360 longitude format — normalize western hemisphere: `lon_folk = lon < 0 ? lon + 360 : lon`
- ⚠️ GFS collection lacks 850 hPa — route 850/925/1000 to `hrrr-isobaric`
- HRRR is US-only but 3km resolution (much better for ag spraying)

**Router logic:** Try Shyft first. On failure (timeout, 5xx, 400), try Folkweather. Return normalized `WindGrid` JSON regardless of source.

### Future: Herbie (Phase 3+)

When Shyft/Folkweather limits bite (rate limits, resolution needs, desire to own the pipeline), add a `HerbieWeatherAdapter` that fetches HRRR directly from AWS S3 via the Herbie library. The adapter pattern means this is a code addition, not a rewrite. Critical HRRR implementation details:

- **Projection:** Lambert Conformal Conic with **spherical earth R = 6,371,229m** — using WGS84 ellipsoid causes positioning errors up to 7km at domain corners.
- **Wind rotation:** HRRR wind components are **grid-relative, not earth-relative**. Must rotate by convergence angle before use:
  ```
  ROTCON = sin(38.5°) ≈ 0.6225
  angle = ROTCON × (lon - (-97.5°)) × π/180
  u_true = cos(angle) × u_grid + sin(angle) × v_grid
  v_true = -sin(angle) × u_grid + cos(angle) × v_grid
  ```
  Shyft/Folkweather EDR APIs return earth-relative winds, so this rotation is only needed for the Herbie adapter.

## 4. Drift Computation

### Phase 1: Ballistic model (TypeScript, runs client-side)

Simple and intentionally crude. A droplet released at height `h` with terminal velocity `v_t` in wind `(u, v)`:

```
fall_time = h / v_t
drift_x = u * fall_time     (east displacement, meters)
drift_y = v * fall_time     (north displacement, meters)
drift_distance = sqrt(drift_x² + drift_y²)
```

Terminal velocity reference values hard-coded by ASABE class (computed via Schiller-Naumann drag at 20°C, sea level — validate these against iterative solver in Phase 2 unit tests):

| Class | VMD (μm) | v_t (m/s) | Re | Drag regime |
|---|---|---|---|---|
| Very Fine | 120 | 0.27 | 0.22 | Stokes-valid |
| Fine | 185 | 0.55 | 0.68 | Stokes-valid |
| Medium | 275 | 0.95 | 1.74 | Schiller-Naumann |
| Coarse | 375 | 1.40 | 3.50 | Schiller-Naumann |
| Very Coarse | 450 | 1.75 | 5.25 | Schiller-Naumann |
| Extremely Coarse | 550 | 2.30 | 8.44 | Schiller-Naumann |

**Note:** These values are initial reference estimates. Phase 2 unit tests must validate each against the iterative Schiller-Naumann terminal velocity solver (converge within 0.5% in ≤ 10 iterations). Any discrepancies → update this table.

Wind at spray height is extrapolated from 10m using the log wind profile:
```
U(z) = U(10m) × ln(z / z₀) / ln(10 / z₀)
```

Surface roughness z₀ by crop type (stored in operation config, validated against Hansen 1993 ARL-TR-61 and WMO 2008):
| Terrain | z₀ (m) | Displacement d (m) | Notes |
|---|---|---|---|
| Bare soil / post-harvest | 0.005 | 0 | Plowed or stubble |
| Short grass / pasture | 0.03 | 0.01 | Mowed turf, hay |
| Low crops (wheat, soy, early corn) | 0.1 | 0.5 | Canopy < 1m |
| High crops (mature corn, sorghum) | 0.25 | 1.5 | Canopy ~2.2m, dense |
| Orchards / vineyards | 0.5 | 2.5 | Trees ~4m |

This runs instantly in the browser. Every time the user scrubs the forecast timeline or changes spray config, drift updates in real time. **Confidence label: "Estimate" — always displayed with the Phase 1 model.**

### Phase 2: Lagrangian particle tracking (Python, server-side)

**Single wind realization:** Track 1,000–5,000 representative particles across 5 ASABE size classes (200–1,000 particles per class) through the current wind field with stochastic turbulence. Uses Schiller-Naumann drag (C_D = 24/Re × (1 + 0.15 × Re^0.687)), RK4 integration with Δt = 0.01s, Gaussian turbulence parameterized by Pasquill-Gifford stability class via the Langevin equation. No evaporation (negligible for droplets ≥ 200μm which dominate drift risk).

Returns P50/P90/P99 drift distance envelopes from the single-realization particle distribution. **Confidence label: "Modeled".**

### Phase 3: Rust physics core (PyO3 + WASM)

Same Lagrangian engine rewritten in Rust. Compiles to:
- Native → PyO3 bindings → replaces Python Tier 2 on server
- WASM → wasm-bindgen → runs in Web Worker for instant client preview

Plus a pre-computed lookup table (50,400 entries, ~800KB) for O(1) queries during timeline scrubbing.

### Phase 3+: Monte Carlo (server only, async)

**Multiple wind realizations:** 100 scenarios, each varying turbulence intensity (within Pasquill-Gifford class bounds), wind speed (±20% measurement uncertainty), and droplet size distribution (random sampling from nozzle DSD). Each scenario runs 1,000 particles across the full size distribution including d²-law evaporation for droplets < 200μm. Total: 100 × 1,000 = 100,000 particle traces. Returns P50/P90/P99 contour polygons from the multi-scenario ensemble. **Confidence label: "Validated".**

## 5. Flight Path Generation

### Algorithm (pure Python + Shapely, ~300–500 lines)

1. **Input:** Field polygon (GeoJSON), wind vector at spray height, drift vector, nozzle swath width, overlap %, buffer distance
2. **Project to UTM** for metric computation (pyproj): `utm_zone = int((lon + 180) / 6) + 1`
3. **Buffer** polygon inward by 3× drone width for headland turning room: `inner = polygon.buffer(-headland_width)`
4. **Compute swath angle** perpendicular to wind: `angle = (wind_dir + 90°) % 360°`
5. **Compute effective swath** = `nozzle_swath - |crosswind_drift|`. If ≤ 0 → return NO_FLY error
6. **Compute line spacing** = `effective_swath × (1 - overlap)`
7. **Generate parallel lines** across the inner polygon at the computed angle and spacing (rotate a bounding box, create evenly spaced horizontal lines, rotate back, clip to inner polygon)
8. **Offset each line upwind** by the drift vector: `translate(line, -drift_x, -drift_y)`
9. **Clip to field boundary** + buffer zone: `line.intersection(inner_polygon)`
10. **Sort lines upwind → downwind** (project line midpoints onto downwind direction vector, sort ascending)
11. **Connect with U-turns** (simple: reverse alternate lines for boustrophedon pattern, add connecting segments)
12. **Add spray on/off waypoints** at line start/end
13. **Reproject to WGS84** for output

### What this doesn't handle (acceptable for MVP)

- Dubins curved turns (simple straight-line U-turns instead)
- Non-convex polygon decomposition (clip-to-polygon handles most cases)
- Battery-optimized swath ordering (boustrophedon is ~90% optimal)
- Fields with holes (subtract holes before step 3 via `polygon.difference(hole)`)

## 7. Client-Side: 3D Wind Dome (Phase 4)

Same concept as the Flutter mobile app. Three.js implementation.

**Rendering architecture:**
- `THREE.Points` with custom `ShaderMaterial` for 4,000–10,000 wind particles (single draw call, trivial on integrated GPUs)
- Per-particle attributes: position, velocity, altitude layer, lifecycle phase
- Altitude color coding computed in fragment shader: green → yellow → orange → red → purple → white
- Ellipsoidal dome as scaled `SphereGeometry` (wireframe, low opacity)
- Flight path as `Line2` (fat lines) with per-vertex spray-on/off colors
- Drift trajectories as second `THREE.Points` system with ballistic shader
- Terrain mesh from Mapbox Terrain-RGB tiles (5–20× vertical exaggeration)
- Camera via `camera-controls` library (smooth damped orbit)

**React integration:** `@react-three/fiber` for declarative scene management. Per-frame animation via `useFrame()` with ref mutation (never React state).

**Performance target:** 60fps on laptop integrated GPU with 10,000 wind particles + 5,000 drift particles + terrain mesh.

## 8. Nozzle Database

No machine-readable public database exists. Boreas curates its own from manufacturer PDF catalogs and university extension publications.

**Schema per nozzle entry:**
```json
{
  "id": "teejet-xr-11002",
  "manufacturer": "TeeJet",
  "model": "XR 11002",
  "type": "flat_fan",
  "spray_angle_deg": 110,
  "asabe_class_at_ref_pressure": "Medium",
  "ref_pressure_bar": 3.0,
  "vmd_um_at_ref_pressure": 275,
  "dv01_um": 130,
  "dv09_um": 480,
  "rosin_rammler_q": 2.4,
  "rosin_rammler_x0_um": 310,
  "flow_rate_lpm_at_ref_pressure": 0.79,
  "pressure_vmd_exponent": -0.3,
  "drone_compatible": true,
  "notes": "Common on DJI AGRAS MG-1, T16"
}
```

**Rosin-Rammler parameters (q, X₀):** Required by the Lagrangian engine (Tier 2+) to sample individual droplet diameters from the nozzle's size distribution. CDF: `F(d) = 1 - exp(-(d/X₀)^q)`. When q and X₀ are not available from published data, compute from Dv0.1/Dv0.5/Dv0.9: solve `0.1 = 1 - exp(-(Dv01/X₀)^q)` and `0.5 = 1 - exp(-(Dv05/X₀)^q)` simultaneously. Typical q values for ag nozzles: **2.0–4.0** (lower = wider distribution = more drift-prone fines).

**Initial scope:** ~30–50 nozzles covering TeeJet XR/AIXR/TT/TTI series (01–04 sizes), DJI centrifugal atomizer models, and Wilger DR/MR/SR tips. Sourced from TeeJet catalog 51A, LSU AgCenter DJI nozzle studies, Purdue extension bulletins.

**Pressure interpolation:** VMD at pressure P = VMD_ref × (P / P_ref)^(−0.3). Exponent varies −0.2 to −0.4 by nozzle type; −0.3 is the standard default.

## 10. Compliance Framework

### Hard-coded label constraints (the primary safeguard)

The pesticide label is the law under FIFRA. Boreas must never recommend parameters that violate label requirements:

- **Wind speed limits:** Warning at > 3 m/s (7 mph). NO FLY at > 4.5 m/s (10 mph). Some labels are stricter.
- **Buffer zones:** Product-specific minimums. If drift + buffer < label minimum → reject.
- **Temperature/inversion:** Wind < 3 mph + stable conditions → inversion risk warning.
- **Application timing:** Some labels restrict to specific times of day.

### Disclaimer language (Climate FieldView pattern)

Every plan display includes: *"Boreas is a decision-support tool. Predictions are estimates based on forecast data and simplified models. Always verify conditions on-site before application. The pesticide label is the law — follow all label requirements. Boreas is not responsible for application outcomes."*

### E&O insurance

Required before commercial launch. ~$67/month for $1M/$1M limits. Essential.

### Spray records

`spray_records` table fulfills federal recordkeeping (7 CFR Part 110): product name, EPA registration number, date/time, area treated, amount applied, wind conditions, applicator license. PDF export for paper filing. 2-year retention minimum for restricted-use pesticides.

## 12. Validation Strategy

### Four-phase approach

**Phase 1 — Unit tests:** Terminal velocity convergence against analytical Stokes solution for droplets < 80μm. Still-air settling time. Uniform crosswind drift (analytical: Δx = V_wind × H / V_t). Mass conservation.

**Phase 2 — AGDISP comparison:** Run identical input scenarios through AGDISPpro trial mode (available from Mount Rose Scientific). Compare deposition curves at 5–300m downwind. Target: R² > 0.8, within 2× of AGDISP predictions.

**Phase 3 — Field data comparison:** Reproduce published deposition curves from the Jerome et al. 114-trial UASS dataset and DJI T30/T25 ISO 22866 studies.

**Phase 4 — Field trial partnership:** Partner with university lab (LSU AgCenter, USDA ARS) for controlled validation. ISO 22866 protocol. Budget: $50–150K per campaign.

### Physics (packages/physics) — Phase 3

Three-crate Rust workspace (core logic separated from platform bindings):

```
packages/physics/
├── Cargo.toml                    # workspace root
├── crates/
│   ├── drift-core/               # Pure Rust: equations, integration, lookup
│   │   ├── Cargo.toml            # deps: nalgebra, rand, rand_chacha, serde
│   │   └── src/
│   │       ├── lib.rs
│   │       ├── droplet.rs        # Terminal velocity, Schiller-Naumann drag
│   │       ├── lagrangian.rs     # Particle tracking engine, RK4 integration
│   │       ├── lookup.rs         # Pre-computed drift table generation + query
│   │       ├── atmosphere.rs     # Log profile, stability, turbulence params
│   │       └── types.rs          # Vec3, DropletClass, DriftResult, WindField
│   │
│   ├── drift-python/             # PyO3 bindings (server target)
│   │   ├── Cargo.toml            # deps: drift-core, pyo3, numpy
│   │   ├── pyproject.toml        # maturin build config
│   │   └── src/lib.rs
│   │
│   └── drift-wasm/               # wasm-bindgen bindings (browser target)
│       ├── Cargo.toml            # deps: drift-core, wasm-bindgen, js-sys
│       └── src/lib.rs
└── tests/
    ├── drift_validation.rs       # Test against AGDISP published values
    └── terminal_velocity.rs      # Test against analytical solutions
```

**Critical:** PyO3 must never be pulled into the WASM build. Use separate Cargo.toml files per crate. `drift-core` uses `#[cfg(not(target_arch = "wasm32"))]` for rayon parallel iteration, falling back to sequential in WASM.

**Floating-point determinism:** Disable FMA on native via `.cargo/config.toml`: `rustflags = ["-C", "llvm-args=-fp-contract=off"]`. Use `rand_chacha::ChaCha8Rng` for deterministic RNG on both targets.

| Crate | Key deps |
|---|---|
| drift-core | nalgebra, rand, rand_chacha, serde, rayon (native only) |
| drift-python | drift-core, pyo3 0.23+, numpy 0.23+ |
| drift-wasm | drift-core, wasm-bindgen, js-sys |

**Build commands:** `cd crates/drift-python && maturin build --release` for Python wheel. `wasm-pack build --target web crates/drift-wasm/` for npm package.
